<objective>
Fix stale-data freeze: add a 2-minute recency window to all 53 latest-value queries across 23 panels so that when inverters go offline at sunset, stat panels show 0/no-data instead of the last active reading.

Purpose: Currently, `ORDER BY time DESC LIMIT 1` with no time filter returns the last recorded value from any point in history. When East/West Microinverters go offline at ~17:00–18:00, all real-time panels (Solar Power, Panel Power, Inverter Power, Roof Layout, Module Detail, Temperature, State/Alarm/Fault) freeze at their last active values all night. This is misleading — the homeowner sees "350W solar production" at 22:00.

Output: `solar-pv-monitor.json` with all 53 affected `rawSql` queries updated to include `WHERE time >= now() - INTERVAL '2 minutes'` before `ORDER BY time DESC LIMIT 1`. The 6 surgical string replacements cover all 53 cases across all 3 measurement sources (East Microinverter, West Microinverter, Smart Meter).
</objective>

## The Stale Data Problem
Inverters report data every ~10 seconds when active. `ORDER BY time DESC LIMIT 1` with no
time filter returns the LAST row ever written — hours-old data after sunset.

## Exact Patterns in solar-pv-monitor.json (as escaped JSON strings)
The file stores SQL inside JSON string values, with `\n` as literal `\\n` escapes.
Six exact patterns cover all 53 stale queries:

### Pattern 1 — East Microinverter (single-line queries, 18 occurrences):
  OLD: `\\"East Microinverter\\" ORDER BY time DESC LIMIT 1`
  NEW: `\\"East Microinverter\\" WHERE time >= now() - INTERVAL '2 minutes' ORDER BY time DESC LIMIT 1`

### Pattern 2 — East Microinverter (multiline queries (panel 23 Module Detail), 4 occurrences):
  OLD: `\\"East Microinverter\\"\\nORDER BY time DESC LIMIT 1`
  NEW: `\\"East Microinverter\\"\\nWHERE time >= now() - INTERVAL '2 minutes'\\nORDER BY time DESC LIMIT 1`

### Pattern 3 — West Microinverter (single-line, 18 occurrences):
  OLD: `\\"West Microinverter\\" ORDER BY time DESC LIMIT 1`
  NEW: `\\"West Microinverter\\" WHERE time >= now() - INTERVAL '2 minutes' ORDER BY time DESC LIMIT 1`

### Pattern 4 — West Microinverter (multiline, 4 occurrences):
  OLD: `\\"West Microinverter\\"\\nORDER BY time DESC LIMIT 1`
  NEW: `\\"West Microinverter\\"\\nWHERE time >= now() - INTERVAL '2 minutes'\\nORDER BY time DESC LIMIT 1`

### Pattern 5 — Smart Meter (single-line, 9 occurrences):
  OLD: `\\"Smart Meter\\" ORDER BY time DESC LIMIT 1`
  NEW: `\\"Smart Meter\\" WHERE time >= now() - INTERVAL '2 minutes' ORDER BY time DESC LIMIT 1`

### Pattern 6 — Smart Meter (multiline, 0 occurrences — include for completeness, no-op):
  OLD: `\\"Smart Meter\\"\\nORDER BY time DESC LIMIT 1`
  NEW: `\\"Smart Meter\\"\\nWHERE time >= now() - INTERVAL '2 minutes'\\nORDER BY time DESC LIMIT 1`

## Exception — Panel 8 (System Status) — DO NOT TOUCH
Panel 8 already has: `WHERE time >= now() - INTERVAL '1 minute' ORDER BY time DESC LIMIT 1`
Because we are inserting WHERE BEFORE the ORDER BY (not replacing ORDER BY),
and Panel 8's FROM clause never has `ORDER BY` immediately after it
(it has `WHERE time...` between FROM and ORDER BY), the patterns above will NOT
match Panel 8 — it is naturally excluded. Verify this after the fix.

## Why 2 minutes?
- Dashboard refresh rate: 10 seconds
- Inverter reporting interval: ~10–30 seconds when active
- 2-minute window: generous enough to handle temporary gaps between readings, tight enough
  to show no-data within ~2 minutes of the inverter going offline at sunset.
- Smart Meter (grid metrics): also gets 2-minute window for consistency, though it rarely
  goes offline.
