# Phase 05.1: Discussion Log

**Phase:** 05.1 — Fix "Today" Data Boundary in SQL Queries
**Mode:** discuss-phase

## Summary

Discussed how to fix all SQL queries in `solar-pv-monitor.json` that use a rolling 24-hour window (`now() - INTERVAL '24 hours'` / `now() - INTERVAL '1 day'`) as a proxy for "today's data", replacing them with a correct calendar-day boundary in Bangkok timezone.

---

## Discussion Transcript

**[Codebase Scout]**
Searched `solar-pv-monitor.json` for all occurrences of rolling-window patterns used as "today" boundaries. Found:
- `now() - INTERVAL '24 hours'` — 16 occurrences across 8 panel groups
- `now() - INTERVAL '1 day'` — 2 occurrences (per-inverter today stats, backfeed panels)
- Total: **18 occurrences** across panels 2, 7, per-inverter today stats, 38, 41, 42, 43, 44, 33, 34

Confirmed NOT affected (already correct):
- `date_trunc('month', now())` — MTD in financial panels
- `date_trunc('year', now())` — YTD in financial panels
- `$__timeFrom` / `$__timeTo` — time-range-aware time-series queries
- `now() - INTERVAL '1 minute'` / `now() - INTERVAL '1 hour'` — real-time readings, not "today" semantics

**[Gray Area Analysis]**
Identified one main decision area: what is the correct SQL expression for "start of today in Bangkok time" when InfluxDB 3 stores timestamps as UTC?

**[Research — InfluxDB 3 Core SQL Docs]**
Fetched `https://docs.influxdata.com/influxdb3/core/reference/sql/functions/time-and-date/` to verify timezone function support.

Confirmed:
- `date_trunc(part, expression)` — supported
- `now()` — returns current UTC timestamp
- `AT TIME ZONE 'zone_name'` — supported for timezone conversion

**[Decision Discussion — D-01: The Fix Expression]**

Proposed expression:
```sql
date_trunc('day', now() AT TIME ZONE 'Asia/Bangkok') AT TIME ZONE 'UTC'
```

Reasoning:
1. `now()` returns current UTC time
2. `now() AT TIME ZONE 'Asia/Bangkok'` converts to Bangkok local time (UTC+7)
3. `date_trunc('day', ...)` truncates to 00:00:00 Bangkok time (midnight Bangkok)
4. Outer `AT TIME ZONE 'UTC'` converts that midnight-Bangkok back to UTC for the `WHERE time >=` comparison
5. Result: always 17:00 UTC of the previous calendar day (= midnight Bangkok), regardless of what time of day it is

Compared to `now() - INTERVAL '24 hours'`:
- At 08:00 BKK, rolling window goes back to 08:00 BKK yesterday — includes yesterday afternoon in "today"
- Calendar fix goes back to 00:00 BKK = 17:00 UTC yesterday — correct day boundary

User confirmed: "you decide" — adopted the researched approach.

**[Scope Confirmation]**

User confirmed: "all panel that currently use `- 24 hours` in query get fix to the 'today' real meaning" — all 18 occurrences, no exceptions.

**[Decision Discussion — D-02: MTD/YTD Unchanged]**

Confirmed that `date_trunc('month', now())` and `date_trunc('year', now())` (already in panels 39/40) are correct enough for practical purposes and are out of scope for this phase.

**[Decision Discussion — D-03: TOU Inner Subquery]**

Noted that for TOU financial panels (38, 41, 42, 43, 44), the `now() - INTERVAL '24 hours'` boundary appears inside a nested subquery. The fix applies to the inner subquery's WHERE clause only — no structural changes to the panel.

**[Outcome]**

All decisions captured in `05.1-CONTEXT.md`. Phase is ready for planning via `/gsd-plan-phase 05.1`.
