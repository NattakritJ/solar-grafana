# Phase 05.1: Fix "Today" Data - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 05.1 fixes all SQL queries in `solar-pv-monitor.json` that use a rolling 24-hour/1-day window (`now() - INTERVAL '24 hours'` or `now() - INTERVAL '1 day'`) to express "today's data". These queries return a sliding 24-hour window instead of the current calendar day in Bangkok timezone.

**In scope:** Every query that uses `now() - INTERVAL '24 hours'` or `now() - INTERVAL '1 day'` as a "today" boundary — 18 occurrences across 9 panel groups.

**Out of scope:**
- Queries using `date_trunc('month', now())` for MTD — already correct
- Queries using `date_trunc('year', now())` for YTD — already correct
- Time-series queries using `$__timeFrom` / `$__timeTo` — already time-range-aware
- Queries using `now() - INTERVAL '1 minute'` or `now() - INTERVAL '1 hour'` — these are for real-time/recent readings, NOT "today" semantics

</domain>

<decisions>
## Implementation Decisions

### The Correct SQL Expression for "Today" (Bangkok Calendar Day)

- **D-01:** Replace `now() - INTERVAL '24 hours'` and `now() - INTERVAL '1 day'` with:

  ```sql
  date_trunc('day', now() AT TIME ZONE 'Asia/Bangkok') AT TIME ZONE 'UTC'
  ```

  **Why this expression:**
  - InfluxDB 3 stores all timestamps as UTC. The `time` column in WHERE clauses is compared against UTC timestamps.
  - Bangkok (Asia/Bangkok) is UTC+7. "Today" in Bangkok starts at 00:00 BKK = 17:00 UTC the previous day.
  - `now() AT TIME ZONE 'Asia/Bangkok'` converts the current UTC time to Bangkok local time.
  - `date_trunc('day', ...)` truncates to midnight Bangkok time.
  - The outer `AT TIME ZONE 'UTC'` converts that midnight-Bangkok timestamp back to UTC for the WHERE comparison.
  - This gives the correct UTC boundary for "start of today in Bangkok" (e.g., at 10:00 BKK, the boundary is 17:00 UTC the prior day).

- **D-02:** The MTD and YTD expressions (`date_trunc('month', now())`, `date_trunc('year', now())`) already exist in the financial panels and are **not changed** — the month/year boundary in UTC aligns closely enough for practical purposes and changing them is out of scope.

### Scope — All Affected Panels

- **D-03:** All 18 occurrences must be fixed. No exceptions. The full list of affected panels and query line numbers:

- **D-04:** For panels 38/41/42/43/44 (TOU financial), the `now() - INTERVAL '24 hours'` appears in the **inner subquery** (the `DATE_BIN` aggregation), not the outer WHERE. The fix is to the inner subquery's WHERE clause only.

### today_production Fields — Leave WHERE Clause as Primary Approach

- **D-05:** Panels 2 and the per-inverter stats query `today_production_N_sensor` fields using `MAX()`. These fields are counters maintained by the inverter hardware that reset at midnight (device-local time). The existing pattern of using `MAX()` over the query window is correct. However, fixing the WHERE clause to `date_trunc('day', ...)` still improves accuracy — it ensures data from after midnight Bangkok is included and data from before midnight (previous calendar day) is excluded.

### Why `now() - INTERVAL '24 hours'` Is Wrong for "Today"
- At 08:00 Bangkok time, `now() - INTERVAL '24 hours'` looks back to 08:00 Bangkok yesterday — including yesterday's afternoon production in "today's" numbers
- The correct boundary: `date_trunc('day', now() AT TIME ZONE 'Asia/Bangkok') AT TIME ZONE 'UTC'` — always returns 17:00 UTC prior day (= midnight Bangkok), regardless of current time of day
